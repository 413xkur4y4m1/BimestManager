Sección 01 · Contexto

Contexto del problema

03 / 14

Las cafeterías universitarias concentran picos de demanda en ventanas muy cortas (entrada, recesos, salida). Esto genera filas, esperas y errores de comunicación, especialmente cuando el cliente tiene alergias o restricciones alimentarias.

En paralelo, los modelos de lenguaje (LLM) y las plataformas low-code de agentes permiten construir interfaces conversacionales que reemplazan formularios y apps móviles dedicadas, con una fracción del esfuerzo tradicional.

El reto es lograr que esta tecnología sea adoptable en entornos educativos que no disponen de licencias premium para conectores personalizados o automatizaciones avanzadas.

Sección 02 · Planteamiento

Planteamiento del problema

04 / 14

¿Cómo ofrecer un canal de pre-pedido conversacional a una comunidad universitaria, integrado a un backend transaccional y desplegable sin licencias Power Platform premium?

01 · A QUIÉN AFECTA

Comunidad universitaria

Estudiantes y personal que consumen en la cafetería del campus. Personas con alergias que requieren comunicación clara con el cafetero.

02 · DÓNDE OCURRE

Punto de venta físico

Cafetería con flujos de hora pico, sin sistema POS conectado a canal digital ni gestión estructurada de notas/alergias por pedido.

03 · LIMITACIONES ACTUALES

Soluciones existentes

Apps móviles dedicadas (Starbucks, Tim Hortons) requieren equipos de desarrollo nativo y presupuesto sostenido — inviables a escala universitaria.

NECESIDAD QUE JUSTIFICA LA PROPUESTA

Se requiere una arquitectura reproducible, basada en OpenAPI estándar y un agente conversacional generativo, que funcione sobre el tenant educativo existente (sin SKUs Power Platform de pago) y exponga las operaciones del backend con seguridad y trazabilidad.

Objetivo general y específicos

OBJETIVO GENERAL

Diseñar e implementar un bot conversacional generativo integrado a un backend Node.js/Express/MySQL, mediante OpenAPI directo en Microsoft Copilot Studio, que permita gestionar el ciclo completo del pedido en una cafetería universitaria sin requerir licencias premium.

Objetivos específicos

OE · 01

Diseñar

El esquema relacional y los 6 endpoints REST que cubren cafeterías, menús, creación, edición, consulta y cancelación de pedidos.

OE · 02

Modelar

El contrato OpenAPI 2.0 con descripciones semánticas que el LLM use para decidir cuándo invocar cada herramienta.

OE · 03

Configurar

El agente en Copilot Studio en modo generativo con prompt de sistema, herramientas API REST y conexión sin autenticación.

OE · 04

Evaluar

El flujo completo end-to-end: listar → ordenar → agregar varios → consultar → cancelar, registrando alergias en notas.

Sección 04 · Fundamentos

Marco teórico

06 / 14
A · IA CONVERSACIONAL

LLM + tool calling

Modelos de lenguaje de gran escala que invocan funciones externas mediante descripciones en lenguaje natural. Base de los agentes modernos (Copilot Studio, ChatGPT Actions, Claude tools).

B · CONTRATOS DE API

OpenAPI 2.0 / Swagger

Especificación neutral que describe rutas, parámetros y respuestas. Permite que un agente generativo descubra capacidades del backend sin acoplamiento manual.

C · ARQUITECTURA WEB

REST · Node/Express · MySQL

Patrón cliente-servidor con recursos uniformes, modelo MVC (Modelo · Vista · Controlador) y persistencia relacional. Estándar de facto en backends transaccionales.

D · LOW-CODE EMPRESARIAL

Microsoft Copilot Studio

Plataforma SaaS para agentes generativos con tópicos, herramientas API REST y canales (Teams, web). Reduce la curva de despliegue de un chatbot a horas en lugar de semanas.

E · EXPOSICIÓN SEGURA

Tunneling HTTPS · ngrok

Dominio reservado desecrate-untwist-smoky.ngrok-free.dev que permite a la nube de Microsoft alcanzar el backend local con TLS, sin abrir puertos en la red institucional.

F · ÁREA DE APLICACIÓN

Hospitalidad inteligente

Subdominio de los sistemas inteligentes orientados al consumo cotidiano: gestión de filas, personalización, accesibilidad alimentaria y reducción de fricción transaccional.

Sección 06 · Método

Metodología

08 / 14

Enfoque iterativo en seis fases — adaptación de CRISP-DM al desarrollo de agentes conversacionales.

FASE 01

Modelado de dominio

Entidades cafetería, menú, producto, pedido. Esquema MySQL y relaciones.

FASE 02

API REST

Implementación MVC en Express: rutas, controladores, modelos. Patrón mysql2 con pool.

FASE 03

Contrato OpenAPI

YAML con 6 operaciones y descripciones orientadas al LLM (cuándo llamar cada una).

FASE 04

Exposición segura

Túnel ngrok con dominio reservado HTTPS, sin abrir puertos institucionales.

FASE 05

Agente Copilot Studio

Modo generativo, 6 herramientas API REST, prompt de sistema con reglas de negocio.

FASE 06

Pruebas E2E

Recorrido completo: descubrir → pedir → agregar → consultar → cancelar.

Contrato compartido

ENTRADAS

Lenguaje natural

Mensajes del usuario en Teams. El LLM extrae intención, entidades (cafetería, productos, cantidades, alergias) y parámetros estructurados.

PROCESAMIENTO

Tool calling generativo

El agente decide cuándo invocar cada endpoint según el contexto conversacional. El prompt de sistema codifica reglas de negocio (un pedido a la vez, notas obligatorias si hay alergia).

SALIDAS

Respuestas + persistencia

Confirmaciones en lenguaje natural + registros en MySQL: pedido creado, productos agregados, estado consultado, pedido cancelado.

Sección 09 · Análisis

Análisis e interpretación

11 / 14
HALLAZGO · 01

El LLM eligió correctamente la herramienta sin reglas explícitas

El modo generativo, alimentado únicamente con descripciones semánticas del OpenAPI, decidió de forma consistente cuándo invocar cada endpoint — confirmando que el diseño de descripciones es más decisivo que las reglas de tópicos clásicas.

HALLAZGO · 02

OpenAPI directo > Custom Connector en tenants educativos

La importación directa de OpenAPI como herramientas "API de REST" evitó completamente la barrera de licenciamiento Power Platform, manteniendo la misma capacidad funcional.

HALLAZGO · 03

El campo notas capturó alergias de forma natural

Al instruir al agente a preguntar por restricciones alimentarias antes de crear el pedido, la información quedó estructurada en MySQL sin necesidad de un formulario explícito — reduciendo fricción de adopción.

HALLAZGO · 04

Implicaciones para el área

El patrón es replicable a otros dominios universitarios: bibliotecas, laboratorios, citas médicas. Cualquier backend REST documentado con OpenAPI se vuelve "conversacionalizable" en horas.

Lectura crítica: los resultados no validan precisión estadística de un modelo entrenado, sino la viabilidad arquitectónica de un patrón de integración. La métrica relevante es la reducción de barrera de entrada, no exactitud/F1.

Sección 10 · Cierre

Conclusiones

12 / 14

01 · Se cumplió el objetivo general: el bot Coffee Nezzi gestiona el ciclo completo de pedido en cafetería universitaria mediante conversación natural.

02 · La integración OpenAPI → Copilot Studio demostró ser una ruta viable para tenants educativos sin licencias premium, eliminando la dependencia de Custom Connectors.

03 · El tool calling generativo permitió eliminar tópicos rígidos: las descripciones semánticas del contrato bastaron para que el LLM orquestara las llamadas.

04 · El patrón es escalable y reproducible: el mismo enfoque aplica a bibliotecas, laboratorios químicos, agenda de citas y cualquier backend REST documentado.

05 · La inclusión del campo notas para alergias mostró que la conversación reduce la fricción en información sensible que un formulario tradicional rara vez captura.

El valor central no es el chatbot — es la arquitectura que lo hace posible en cualquier institución educativa.

Sección 11 · Cierre extendido

Aportaciones · Limitaciones · Trabajo futuro

13 / 14
APORTACIONES

Lo que ofrece

  • Patrón reproducible OpenAPI → Copilot Studio sin premium.

  • Manejo estructurado de alergias por conversación.

  • Documentación operativa replicable a otras facultades.

  • Demostración de canal Teams como punto de venta digital.

LIMITACIONES

Lo pendiente

  • Conexión "Sin autenticación" no es apta para producción.

  • ngrok-free expone un único túnel, sin alta disponibilidad.

  • Sin métricas de uso ni telemetría conversacional.

  • Evaluación piloto cualitativa, sin estudio cuantitativo amplio.

TRABAJO FUTURO

Próximos pasos

  • Auth con API Key + rate limiting en el backend.

  • Despliegue Cloud Run / Azure App Service con dominio propio.

  • Integración con pasarela de pago (Mercado Pago/Stripe).

  • Análisis de logs conversacionales para detectar patrones de demanda.
