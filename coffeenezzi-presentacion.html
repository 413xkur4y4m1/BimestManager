01 · ¿Qué hace?

Un asistente que toma tu pedido de café por chat.

Coffee Nezzi es un agente conversacional construido en Microsoft Copilot Studio, conectado a un backend Node.js + MySQL. El usuario chatea en lenguaje natural y el bot resuelve todo el flujo: elige cafetería, muestra el menú, arma el pedido, calcula total y le devuelve número de orden.

Conversación natural

El usuario escribe "quiero un latte grande con avena" y el LLM de Copilot Studio extrae los parámetros y llama a la herramienta correcta del backend.

Pedido en MySQL

Cada conversación se traduce a un registro real en la base. Tabla pedidos, detalle_pedido y stock decremental en menus.

Display en vivo

En la barra de la cafetería corre una pantalla en EJS + Socket.IO que muestra pedidos en cola en tiempo real. Sin refrescar.

Demo conversación

Así suena una orden real.

El bot no devuelve menús desplegables ni formularios. Mantiene una conversación humana, llama a la API cuando hace falta y guarda el pedido en MySQL.

02 · ¿Qué soluciona?

Las filas de la cafetería no son problema de café. Son problema de orden.

En cafeterías escolares y de oficina, los 15 minutos de descanso se gastan formado. El alumno no quiere descargar otra app ni registrarse. El cafetero no quiere otra tablet con otra interfaz.

Sin Coffee Nezzi

El día a día actual

  • ×Filas de 10–15 minutos en cada receso
  • ×Confusiones de nombres en la barra ("¿quién pidió el americano?")
  • ×Sin registro de qué se vendió ni cuándo
  • ×Stock controlado a ojo, sin datos para reabastecer
  • ×Apps caras o suscripciones SaaS para algo simple
Con Coffee Nezzi

Lo que cambia

  • ✓El alumno pide por chat antes de salir del aula
  • ✓El pedido aparece con número y nombre en la pantalla de la barra
  • ✓Cada venta queda en MySQL — datos reales para el dueño
  • ✓Stock se decrementa solo cuando se agrega producto
  • ✓Cero apps. Cero suscripciones premium. Cero fricción.

04 · Procesos

Seis verbos. Una orden completa.

El bot tiene seis herramientas para cubrir el flujo end-to-end. Cada una con descripción explícita para que el LLM sepa cuándo invocarla.

01
GET /api/cafeterias

Listar cafeterías

El bot descubre dónde puede ordenar el usuario.
// Salida
[{ id: 1, nombre: "La Cafetera" },
 { id: 2, nombre: "Café del Patio" }]
02
GET /api/menus/{cafeteria_id}

Obtener menú

Productos disponibles agrupados por categoría con stock vigente.
// Entrada · Salida
cafeteria_id: 1 → [{ nombre: "Latte", precio: 45.00, cantidad: 12 }, ...]
03
POST /api/pedidos

Crear pedido

Inicia una orden a nombre de una persona. Acepta notas/alergias.
// Body
{ cafeteria_id: 1, nombre_persona: "Daniel", notas: "sin azúcar" }
→ { pedido_id: 127 }
04
POST /api/pedidos/{id}/productos

Agregar producto

Suma items al pedido. Valida stock, calcula subtotal, decrementa cantidad.
// Body · validaciones
{ menu_id: 5, cantidad: 2 }
✗ "Stock insuficiente" · ✗ "Producto no existe"
05
GET /api/pedidos/{id}

Consultar estado

Devuelve estado actual: recibido → preparación → listo → entregado.
// Salida
{ id: 127, total: 75.00, estado: "preparacion" }
06
PUT /api/pedidos/{id}/cancelar

Cancelar pedido

Marca el pedido como cancelado. Stock se restaura por trigger.
// Acción reversible
estado: "recibido" → "cancelado"
// Devuelve el stock al inventario

Tres roles, tres vistas.

Además del bot, el backend sirve dos paneles EJS para el equipo operativo.

/admin
Admin
Da de alta cafeterías, asigna roles, ve métricas globales.
/cafetero
Cafetero
Gestiona menú y stock, actualiza estado de pedidos en su barra.

Display público
Pantalla con cola de pedidos en vivo, vía Socket.IO. Se cuelga en la barra.

Decisión

OpenAPI directo → no Custom Connector

El tenant educativo de La Salle no expone Custom Connectors premium. Se importa el openapi.yaml directamente en Copilot Studio como API de REST. Cero licencias premium.

Decisión

mysql2 con decimalNumbers: true

Sin esa flag, Power Fx (motor de Copilot Studio) rompe con PowerFxJsonException al recibir DECIMAL. Pool configurado para devolver números nativos.
