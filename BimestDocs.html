DOCUMENTACIÓN OFICIAL

BimestManager

Plataforma web para la gestión integral de laboratorios escolares — módulos de Química y Turismo — con control de inventario, préstamos, adeudos, incidencias, firmas digitales y monitoreo en tiempo real.

Stack: Node 20 · Express 5 · MySQL 8 Patrón: MVC Seguridad: JWT + HTTPS + TLS Licencia: ISC

Procesos

Cómo se desarrollan los flujos de negocio dentro de la plataforma, desde que un usuario abre la app hasta que se registra una devolución de material.

Flujo general del usuario

1
Acceso a /login
El cliente se redirige a la vista EJS de login. El navegador establece conexión https:// contra el puerto 4443.
2
POST /auth/login
El controlador valida credenciales contra MySQL, compara el hash bcrypt y firma un JWT (HS256, 2h).
3
Cookie httpOnly
Se guarda bimest_token con sameSite=lax y secure en producción. El JS del cliente no puede leerla.
4
Redirección por rol y fuente
El middleware obtenerRedirectPorRol() envía al usuario a su panel: /admin, /maestro, /estudiante, /turismo/admin o /turismo/estudiante.
5
Llamadas a API protegida
Cada request a /admin/*, /maestro/*, etc. pasa por autenticar → cargarUsuarioActual → autorizarFuentes → autorizarRoles.
6
Logout
POST /auth/logout ejecuta limpiarCookieSesion() y el JWT queda inutilizable en este navegador.

Proceso laboratorio Química

Roles involucrados: Administrador, Maestro, Alumno.

1
Admin carga inventario
Da de alta materiales, define prácticas y arma kits (materiales + cantidades) desde /admin.
2
Maestro programa sesión
Crea una sesión asociada a un grupo y a una práctica. Forma equipos y asigna alumnos.
3
Alumno firma responsiva
El alumno entra, lee el reglamento y firma con el dedo/mouse. La firma viaja como dataURL base64 y se guarda en imageFirma/ + tabla responsivas.
4
Préstamo de material
El admin registra el préstamo del kit al equipo. Stock se descuenta en materiales.
5
Cierre y devolución
Al devolver, se marca devuelto_en. Si hay faltantes se crea un adeudo y/o una incidencia.

Proceso laboratorio Turismo

Diferencia clave: en Turismo el alumno puede solicitar el préstamo y el admin lo aprueba o rechaza, mientras que en Química el admin registra directamente.

1
Solicitud del alumno
POST /turismo/estudiantes/prestamos crea un préstamo en estado pendiente.
2
Aprobación
El admin de Turismo aprueba (PATCH /turismo/admin/prestamos/:id/aprobar) o rechaza. Se notifica al alumno.
3
Sugerencias de práctica
El admin agrega sugerencias de material recomendado por práctica para guiar al alumno.
4
Devolución / adeudo
El admin marca devuelto o adeudo. Las notificaciones leida=false se acumulan en la campanita del alumno.

Proceso de monitoreo

Cada vez que se solicita GET /monitor/metricas, el controlador monitoreoControlador.js:

  • Calcula porcentaje de CPU promediando los tiempos de os.cpus() y refina con systeminformation.
  • Lee RAM con os.totalmem() / freemem().
  • Pide disco y red con si.fsSize() y si.networkStats().
  • Hace ping al pool de MySQL y mide latencia.
  • Devuelve un JSON estructurado consumible por Prometheus/Grafana o por el dashboard /monitor.

Arquitectura

BimestManager sigue un patrón MVC estricto sobre Node.js, con frontera limpia entre rutas, controladores y modelos. El despliegue se diseñó para correr en una topología de cuatro servidores aislados.

Visión general

El cliente (navegador o móvil) conversa con el servidor WEB vía HTTPS. WEB ejecuta Express, renderiza vistas EJS y expone una API REST. WEB se conecta a DB (MySQL 8) por TLS, autenticado contra DC (Active Directory), y publica métricas hacia MON.

Patrón MVC

rutas/<area>Ruta.js ──▶ controlador/<area>Controlador.js ──▶ modelo/<area>Modelo.js ──▶ MySQL (Express) (lógica HTTP, sin SQL) (única capa con SQL)

Regla de oro: los controladores nunca hablan directo con MySQL. Toda consulta vive en modelo/. Esto permite cambiar el motor o agregar caché sin tocar la lógica HTTP.

Topología 4 servidores (DC / WEB / DB / MON)

┌───────────────────────────┐ │ DC (Active Directory) │ │ DNS · Cuentas · GPO │ │ 192.168.10.10 │ └─────────────┬─────────────┘ │ ┌─────────────────────┼─────────────────────┐ │ │ │ ┌────────▼────────┐ ┌────────▼────────┐ ┌────────▼────────┐ │ WEB │ │ DB │ │ MON │ │ Node 20 + IIS │ │ MySQL 8 + TLS │ │ Prometheus / │ │ HTTPS 4443 │ │ schema labs │ │ Grafana │ │ 192.168.10.20 │ │ 192.168.10.30 │ │ 192.168.10.40 │ └────────┬────────┘ └────────▲────────┘ └─────────────────┘ │ │ └──── TLS :3306 ──────┘
Domain Controller
Active Directory, DNS interno, políticas GPO. Centraliza cuentas del personal.
Web Server
Node 20 + Express 5 detrás de IIS con ARR como reverse proxy. Sirve HTTPS 4443.
Database
MySQL 8 con require_secure_transport=ON. Sólo acepta TLS.
Monitoring
Prometheus + Grafana scrapean /monitor/metricas cada 15s.

Seguridad

Modelo de defensa en profundidad: el atacante tiene que vencer varias capas independientes antes de tocar datos sensibles.

Capas implementadas

Transporte
HTTPS obligatorio en producción. Cert generado con selfsigned en dev o emitido por CA en prod. HSTS 1 año, includeSubDomains.
Cabeceras HTTP
helmet activo: CSP, X-Frame-Options, X-Content-Type-Options. Se deshabilita el header x-powered-by.
Autenticación
JWT firmado con HS256, expiresIn=2h. Secret obligatorio en .env (JWT_SECRET).
Sesión
Cookie bimest_token httpOnly, sameSite=lax, secure en prod. JS del cliente no puede leerla → mitiga XSS.
Contraseñas
Hash con bcrypt. Nunca se almacenan en claro ni viajan más allá del controlador.
Autorización
RBAC granular por rol (ADMIN, MAESTRO, ALUMNO) y por fuente (QUIMICA, TURISMO). Un admin de Química no puede tocar Turismo.
Base de datos
TLS forzado (require_secure_transport=ON). Usuario bimest_app con sólo SELECT/INSERT/UPDATE/DELETE.
SQL Injection
Todo query usa mysql2 con parámetros preparados (placeholders ?). Nada de concatenación de strings.
Auditoría
Winston con rotación diaria escribe app-YYYY-MM-DD.log, error-YYYY-MM-DD.log, access-YYYY-MM-DD.log.

Autenticación JWT — detalle

El middleware autenticar() extrae el token de la cookie bimest_token o del header Authorization: Bearer .... Lo verifica con jwt.verify() y, si está expirado o inválido, limpia la cookie y responde 401.

Acto seguido, cargarUsuarioActual() consulta el resumen del usuario en la BD para confirmar que sigue existiendo y que is_active=1. Esto permite revocar accesos en caliente desactivando una cuenta.

TLS / HTTPS

En el primer arranque, Middlewares/certificados.js verifica si existen certs/server.key y certs/server.crt. Si no, los genera con selfsigned: RSA 2048 bits, SHA-256, válido 365 días, con SAN para localhost, 127.0.0.1 y opcionalmente IPs adicionales vía HTTPS_SAN_IPS. La key se guarda con permisos 0600.

Base de datos

El schema labs agrupa tanto las tablas de Química como de Turismo (con sufijo _turismo). Toda conexión usa pool con TLS opcional.

Motor y conexión

Motor: MySQL 8. Driver: mysql2/promise. Pool: 10 conexiones, sin límite de cola. Si DB_SSL=true, se construye el bloque SSL leyendo certs/mysql-ca.pem o el path indicado en DB_SSL_CA. DB_SSL_REJECT_UNAUTHORIZED permite aceptar certs autofirmados en desarrollo.

Variables: DB_HOST, DB_PORT=3306, DB_USER, DB_PASSWORD, DB_NAME=labs.

Roles MySQL

El script sql/usuario_app.sql crea un usuario dedicado bimest_app con permisos mínimos DML (SELECT, INSERT, UPDATE, DELETE) sobre el schema labs. Nunca se usa root en producción. El usuario root queda reservado para migraciones y mantenimiento administrativo.

Modelo OSI

Cada componente de BimestManager se ubica en una capa específica del modelo de referencia OSI. Esto ayuda a razonar fallas y planear seguridad por capas.

Mapeo por capa

7
AplicaciónApplication
Vistas EJS, panel admin/maestro/alumno, dashboard /monitor. Endpoints REST /auth, /admin, /maestro, /turismo/*.
6
PresentaciónPresentation
JSON (express.json), HTML, CSS. Codificación UTF-8. Cifrado TLS 1.2/1.3 negociado por https de Node. Firmas guardadas como dataURL base64.
5
SesiónSession
JWT firmado con HS256 (jsonwebtoken) + cookie bimest_token httpOnly. Pool MySQL mantiene 10 sesiones persistentes.
4
TransporteTransport
TCP en puertos 4000 (HTTP), 4443 (HTTPS) y 3306 (MySQL). Sin UDP en el flujo principal.
3
RedNetwork
IPv4 en LAN 192.168.10.0/24 (DC .10, WEB .20, DB .30, MON .40). Ruteo interno + DNS resuelto por DC.
2
Enlace de datosData Link
Ethernet entre servidores, VLAN aislada para tráfico de DB. ARP estándar.
1
FísicaPhysical
Cableado RJ-45 Cat 6 / fibra entre rack y switches del laboratorio. UPS recomendado para WEB y DB.

Clonar e instalar

Pasos completos desde cero para tener el proyecto corriendo en local:

# 1. Clonar el repositorio
git clone https://github.com/413xkur4y4m1/BimestManager.git
cd BimestManager

# 2. Instalar dependencias (Node 20+ y npm 10+)
npm install

# 3. Configurar variables de entorno
cp .env.example .env
# Windows PowerShell:  Copy-Item .env.example .env

# 4. Cargar el schema y los seeds en MySQL
mysql -u root -p < creacionTabs.sql
mysql -u root -p labs < labs_usuarios.sql

# 5. Levantar el servidor
npm run dev     # desarrollo con nodemon
npm start       # produccion

# 6. Abrir en el navegador
# HTTP:  http://localhost:4000
# HTTPS: https://localhost:4443

Flujo de ramas

main
Rama estable. Solo recibe merges desde develop tras revisión. Cada merge genera un release etiquetado.
develop
Integración continua. Aquí confluyen las feature/* antes de pasar a main.
feature/*
Una rama por funcionalidad nueva. Ej: feature/turismo-sugerencias, feature/dashboard-monitor.
fix/*
Correcciones puntuales de bugs detectados en pruebas o producción.
hotfix/*
Parches urgentes que se aplican directo sobre main y luego se sincronizan a develop.

feat(turismo): agregar aprobacion de prestamos
fix(auth): corregir expiracion de cookie en HTTPS
docs(readme): actualizar guia de despliegue
chore(deps): bump express a 5.2.1
refactor(modelo): extraer queries de prestamos a helper

Cómo contribuir

1. Haz fork del repo y crea una rama feature/<nombre> a partir de develop.
2. Asegúrate de que el servidor arranca limpio (npm run dev) y que no hay logs de error.
3. Sigue el patrón MVC: nada de SQL en controlador/, ningún req/res en modelo/.
4. Abre un Pull Request hacia develop describiendo el cambio y referenciando el issue.
5. Espera revisión de al menos un mantenedor antes del merge.
